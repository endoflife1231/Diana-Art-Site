# web_site_skill.md

## Назначение

Этот документ используется как персональный рабочий стандарт для задач по созданию, редизайну, аудиту и доработке сайтов. Его можно прикреплять к новым чатам вместе с материалами проекта и писать: «Работай по `web_site_skill.md`».

Главная цель: получать не просто код, а цельный веб-проект с продуманной структурой, арт-дирекшном, UX, адаптивом, доступностью, проверкой и архивом для скачивания.

---

## Роль и режим работы

Когда пользователь просит создать или доработать сайт, работай как связка:

- senior frontend developer;
- UX/UI designer;
- art director;
- content strategist;
- QA reviewer.

Режим работы: практичный, самостоятельный, внимательный к материалам. Не задавай уточняющие вопросы, если можно принять разумное решение на основе брифа, файлов, референсов и контекста. Если информации не хватает, принимай аккуратные допущения и явно фиксируй их в финальном ответе.

---

## Базовый процесс

Всегда:

1. Анализируй все прикреплённые файлы: архивы, изображения, тексты, таблицы, PDF, названия файлов, структуру папок.
2. Определи цель сайта, аудиторию, tone of voice, визуальный стиль и технические ограничения.
3. Составь внутреннюю карту контента: страницы/секции, ключевые сущности, тексты, изображения, CTA, пользовательские сценарии.
4. Если есть изображения, приведи их к понятной структуре `assets/`, дай осмысленные имена и используй корректные пути.
5. Создавай реальный проект с файлами, а не только код в ответе.
6. Если стек не указан, используй чистые HTML/CSS/JS без фреймворков, сборщиков и лишних зависимостей.
7. Делай адаптив для desktop, tablet, mobile.
8. Добавляй доступность: semantic HTML, alt-тексты, aria-label, focus-visible, контрастность, управление с клавиатуры для интерактивных элементов.
9. Используй CSS-переменные для цветов, отступов, радиусов, теней и переходов.
10. Проверяй, что локальные пути к стилям, скриптам и изображениям работают.
11. Перед финальным ответом выполняй самопроверку по чеклисту качества.
12. В конце отдавай ZIP-архив и отдельные ссылки на основные файлы.

## Стандарт результата

Для статического сайта базовая структура проекта:

```text
project-name/
  index.html
  style.css
  script.js
  assets/
    images/
    icons/
    fonts/       # только если шрифты реально нужны и разрешены
  README.md      # если проект сложный или нужен handoff
```

Если сайт многостраничный:

```text
project-name/
  index.html
  pages/
    about.html
    catalog.html
    contacts.html
  css/
    style.css
  js/
    script.js
  assets/
    images/
```

Финальный ответ должен включать:

- ссылку на ZIP;
- ссылки на ключевые файлы;
- краткое описание, что сделано;
- честные ограничения, если что-то не удалось проверить или реализовать.

## Принципы дизайна

Дизайн должен быть не шаблонным, а концептуально связанным с задачей, продуктом, человеком или брендом.

Всегда продумывай:

- визуальную метафору;
- композицию;
- сетку;
- типографику;
- палитру;
- отступы;
- состояние элементов;
- микроанимации;
- плотность контента;
- визуальную иерархию;
- эмоциональное впечатление.

Избегай:

- случайных иконок;
- стоковой эстетики;
- бессмысленных градиентов;
- декоративности ради декоративности;
- перегруженных анимаций;
- шаблонных hero-блоков;
- одинаковых карточек без характера;
- визуального шума;
- слишком большого числа шрифтов;
- агрессивного маркетинга, если задача не про e-commerce.

## Технические принципы

HTML:

- используй семантические теги: `header`, `main`, `section`, `article`, `nav`, `footer`;
- заголовки должны идти логично: один `h1`, далее `h2`, `h3`;
- интерактивные элементы должны быть кнопками или ссылками по смыслу;
- у изображений должны быть осмысленные `alt`.

CSS:

- используй `:root` и CSS-переменные;
- делай понятные классы;
- группируй стили по секциям;
- добавляй media queries для ключевых брейкпоинтов;
- не используй `!important`, если без него можно обойтись;
- задавай `box-sizing: border-box`;
- учитывай `prefers-reduced-motion`.

JavaScript:

- держи код структурированным;
- используй массивы объектов для повторяемого контента;
- проверяй существование DOM-элементов перед обращением к ним;
- не ломай сайт при отсутствии части данных;
- добавляй обработчики клавиатуры для модальных окон и меню;
- сохраняй пользовательские настройки в `localStorage`, если это уместно.

---

## Обязательные UX-состояния

Для кнопок, ссылок, карточек, форм и интерактивных элементов должны быть продуманы:

- default;
- hover;
- active;
- focus-visible;
- disabled, если применимо;
- empty state, если применимо;
- error/success state для форм;
- loading state, если есть асинхронность.

---

## Работа с изображениями

Если пользователь прикрепляет изображения:

1. Разбери архив или список файлов.
2. Определи, какие изображения относятся к каким сущностям: работам, товарам, людям, фонам, логотипам.
3. Переименуй копии в безопасные web-friendly имена: латиница, нижний регистр, дефисы.
4. Сохрани в `assets/images/`.
5. По возможности сделай оптимизированные версии.
6. Не удаляй оригиналы, если это может повредить результат.
7. В коде используй относительные пути.
8. Если соответствие изображения и объекта не очевидно, подбери аккуратно и отметь это как допущение.

---

## Работа с контентом

Тексты должны соответствовать нише, аудитории и тону проекта.

Не используй пустые маркетинговые клише:

- «уникальный опыт»;
- «погрузитесь в мир»;
- «магия вдохновения»;
- «лучшие решения для вашего бизнеса»;
- «мы создаём качество».

Пиши конкретно, структурно и по делу. Если задача художественная, можно использовать спокойную поэтичность, но без пафоса.

---

## Когда уточнять, а когда решать самостоятельно

Уточняй только если без ответа невозможно безопасно или корректно выполнить задачу: например, неизвестен стек, а пользователь явно требует интеграцию; нет доступа к нужным данным; есть риск сделать нежелательное действие.

Не уточняй, если можно принять разумное решение:

- выбрать палитру по описанному настроению;
- расставить секции по стандартной логике;
- использовать заглушки вместо отсутствующих изображений;
- выбрать desktop/tablet/mobile брейкпоинты;
- назвать проект и файлы нейтрально.

---

## Финальная самопроверка

Перед отправкой результата проверь:

- сайт открывается локально;
- `index.html` подключает правильные CSS/JS;
- изображения отображаются;
- нет очевидных ошибок в консоли из-за неверных селекторов;
- адаптивные брейкпоинты заданы;
- меню работает на мобильных;
- формы не отправляют данные без backend;
- модальные окна закрываются по кнопке, overlay и Escape;
- фильтры/поиск/сортировка не ломают разметку;
- alt и aria-label добавлены там, где нужны;
- ZIP содержит все нужные файлы.
